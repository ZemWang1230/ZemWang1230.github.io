# 读懂 Vitalik 的 P/N 命题

六月初，Vitalik在 ETHResearch 上发布了一篇帖子：[https://ethresear.ch/t/building-index-tracking-assets-on-top-of-options-instead-of-debt/25036](https://ethresear.ch/t/building-index-tracking-assets-on-top-of-options-instead-of-debt/25036)；

讨论了用`期权`而不是债务构建指数追踪资产；

## 这篇文章真正讨论的是什么？

> 比如，当一个链上系统只能持有ETH，却想让用户获得美元、商品价格或其他外部指数的敞口时，该系统怎么应对和处理极端行情下的价格变化？

目前传统的 DeFi 的默认做法是债务和清算；

用户抵押ETH，协议 mint 某种稳定资产或合成资产。当ETH价格下跌，且抵押品价值不足时，协议通过清算机制把抵押品卖掉，避免系统产生坏账；

所以 Vitalik 就想，能不能不设计一个更快、更狠、更复杂的清算系统，而是从一开始就设计一种不需要清算的结构？

现在目前的 DeFi 设计：

```text
有合成资产 -> 有债务 -> 有清算机制 -> 有实时的预言机
```

现在想打断这些默认机制：

合成资产不一定产生债务，没有了债务，就不一定需要清算，没有清算，预言机就不必实时决定用户的生死；

## 为什么系统只能持有 ETH 是一切问题的起点

在一个纯链上系统中时，唯一真正无需信任的抵押资产就是native token（也就是ETH）；

但是光有ETH没办法与现实世界交互，所以，这就导致了用户可能需要：

- USD 稳定敞口
- 黄金的价格敞口
- 某只股票的价格敞口
- ...

这就导致出现了一个问题，系统无法凭空创造出真实的美元，也不能凭空创造一个股票或商品的价格，它只能再分配ETH的风险和收益；

> 比如，用户A想获得一些`正向USD敞口`，那么系统里必须有用户B承担对应的`负向USD敞口`，否则这个敞口就没有来源；

可以理解为：系统内部关于指数 T 的总敞口 == 0；

> 如果系统中只有ETH，却有人想要稳定美元价值，那么稳定性不是凭空产生的，肯定有另一方愿意承担 ETH/USD 变化带来的相反风险；

## 为什么债务模型会自然走向清算？

先来看看最熟悉的CDP（Collateralized Debt Position，抵押债仓）模型：用户把资产抵押进去，然后借出协议发行的稳定币；

假设当前ETH的价格是1500美元，用户抵押 1 ETH，借出1000美元的稳定币；

此时合约中记录的就是：

```text
抵押品：1 ETH
债务：1000 USD
```

当ETH的价格为2000美元时，一切都没有问题：

```text
抵押品价值 = 2000美元
债务价值 = 1000美元
抵押率 = 2000 / 1000 = 200%
```

但如果ETH由于极端行情，突然跌到900美元：

```text
抵押品价值 = 900美元
债务价值 = 1000美元
```

系统中已经资不抵债了；所以协议必须在更早的时候触发清算，比如跌到1100美元的时候就触发清算，强制把用户的ETH卖掉，用来偿还债务。

债务模型的问题在于，它承诺了一个固定的负债--"无论ETH怎么变，你都欠1000美元"，但是抵押品价值不是固定的，当一个固定负债遇到一个波动抵押品时，清算是不可能避免的。

## 清算不是附加功能，而是债务模型的生存条件

其实，从设计机制上来看，清算是债务系统的必要组件，没有清算机制，债务系统就会遇到很多问题：

- 坏账：如果抵押品价值低于债务价值，系统无法全额偿还稳定币持有人，导致系统亏损；
- 挤兑：如果用户发现系统有坏账，就会抢先退出，留下最后的人承担损失；
- 死亡螺旋：清算失败会削弱信心，信心下降会让稳定资产脱锚，脱锚又会放大系统损失；
- ...

所以债务型系统必须在抵押品还足够时提前清算，这也是为什么Aave、Maker等知名协议都有复杂的抵押率、清算线、拍卖和keeper激励等等机制；

这些机制其实都在处理同一个问题：如何在抵押品不足之前，把风险头寸强制关闭；

## 清算为什么依赖实时预言机

清算系统要回答一个问题：

> 这个用户现在是否已经危险到必须被清算的程度？

首先，必须得是实时的，如果ETH正在快速下跌，协议不能等几分钟后再知道价格，几分钟后，抵押品的价值可能远远不够负债了；

其次，必须有强制执行力，清算并不是简单的参考价格，而是会直接改变用户资产归属的强制动作，预言机给出的价格一旦触发清算，用户的头寸就会被关闭；

还有，清算往往发生在暴跌、拥堵、流动性紧张、MEV活跃的时候，也就是说，预言机最需要可靠的时候，恰恰是环境最恶劣的时候；

这也就是Vitalik认为实时预言机比较危险的原因，实时预言机的问题包括：

- 可被短期价格操纵；
- 很难加入争议期，因为清算需要立刻执行；
- 依赖少数自动化数据提供者；
- 在极端行情中可以延迟、拥堵、失真或者宕机；
- 一旦被错误触发，会造成不可逆的用户损失；

相反，慢预言机可以做很多安全增强，例如争议、仲裁、多源验证、预测市场的前置发现等等；但目前的状况就是，实时清算预言机无法充分利用这些机制，因为它没有时间。

## 不要优化清算，而是移除清算

之前也说到，CDP模型的思路是：

````text
实时预言机不安全 -> 做更强的预言机
清算可能失败 -> 做更快、更稳的清算
坏账风险存在 -> 提高清算线
``

Vitalik提出一个新的思路：

```text
为什么一定要有清算？ -> 为什么一定要有债务？ -> 能否让系统永远不会欠超过抵押品的东西？
````

如果协议从一开始就不承诺固定美元债务，而只是承诺"根据到期价格分配已有ETH"，那么系统就不会出现抵押品不足；

换句话说，风险并不是消失了，而是被重新表达了；

在债务系统中，风险表现为：

```text
抵押品价值 < 债务价值
```

在新思路中，风险表现为：

```text
用户持有的资产不再精确追踪目标指数
```

前者需要清算，后者需要再平衡。

## P/N 结构是什么

Vitalik定义了两个资产，一个`P`，一个`N`；

他们由 1 ETH 拆分得到：

```text
1 ETH -> 1 P + 1 N
```

每一个 P/N 系列有三个参数：

- `T`：要追踪的指数，例如 USD/ETH
- `S`：strike，行权价
- `M`：maturity，到期日

到期时，预言机给出`T`的值，记为`x`；

结算的规则是：

```text
P = min(1, S / x) ETH
N = max(0, 1 - S / x) ETH
```

这样无论x是什么值，算下来`P + N = 1 ETH`；

这其实也就意味着，`P`和`N`只是用一份 ETH 抵押品的两种互补索取权，它们之间的分配比例会变化，但系统总支付不会超过1ETH。

## 用例子理解 P/N 的稳定敞口

假设我们想构造接近USD的稳定敞口；

设：

```text
S行权价 = 1500
x = ETH/USD的到期价格
```

如果到期时ETH/USD = 3000:

```text
P = min(1, 1500 / 3000) = 0.5 ETH
N = max(0, 1 - 0.5) = 0.5 ETH
```

`P`的美元价值是：`0.5 ETH * 3000 USD/ETH = 1500 USD`；

如果到期时ETH/USD = 2000:

```text
P = min(1, 1500 / 2000) = 0.75 ETH
N = max(0, 1 - 0.75) = 0.25 ETH
```

`P`的美元价值是：`0.75 ETH * 2000 USD/ETH = 1500 USD`；

如果到期时ETH/USD = 1500:

```text
P = min(1, 1500 / 1500) = 1 ETH
N = max(0, 1 - 1) = 0 ETH
```

`P`的美元价值是：`1 ETH * 1500 USD/ETH = 1500 USD`；

通过上面三个例子，可以看到，只要ETH/USD高于1500，`P`的美元价值就接近1500美元；这就是为什么`P`可以被用来构造稳定敞口的原因。

## 为什么 P 不是完美稳定币

上面说了，只要ETH/USD高于1500，就很稳定；

如果到期时，ETH/USD = 1000:

```text
P = min(1, 1500 / 1000) = 1 ETH
N = max(0, 1 - 1) = 0 ETH
```

`P`最多只能拿到 1ETH；

但是注意，此时的1 ETH只值1000美元，所以`P`的美元价值变成了`1 ETH * 1000 USD/ETH = 1000 USD`；

这其实不是bug，而是系统一开始就约束的结果：

> 系统里只有1ETH，它不可能在ETH跌到1000美元时支付1500美元的价值；

所以 P/N 结构只能在某些价格区间提供稳定敞口，一旦ETH跌破strike，稳定性会下降；但是要区分，这个下降不是清算，而是追踪偏离。

## 清算风险与漂移风险的区别

债务模型的用户体验通常是：

```text
看起来稳定 -> 接近清算线 -> 被强制清算
```

它的风险有明显断点；在断点之前，用户感觉自己持有稳定资产。断点之后，头寸被强制关闭。

P/N 模型的用户体验是：

```text
比较稳定 -> 稳定性逐渐变差 -> 用户选择是否再平衡
```

它没有一个强制清算的断点，风险从"协议突然对用户执行动作"变成"用户的目标敞口逐渐偏离"。

这不是说 P/N 风险更小，而是说风险形态更连续、更可观察、更适合用户自主决策。

## 为什么慢预言机在 P/N 里可行

P/N 系统不需要实时判断谁该被清算，它只需要在到期时知道：

```text
T(要追踪的指数) 在 M(到期日) 时的值是多少？
```

这个问题不需要即时的回答，它可以：

- 延迟几个小时；
- 允许提交者提出结果；
- 允许其他人挑战结果；
- 在有争议时调用更昂贵的裁决机制；
- 使用预测市场作为前置价格发现层；
- ...

这些都是慢预言机的优势，慢预言机不是"慢所以差"，而是"慢所以有时间做安全"；

在实时清算场景中，如果 oracle 价格错了，用户可能已经被清算，损失难以恢复。

在到期结算场景中，如果 oracle 结果有争议，可以先暂停结算，让争议机制运行。

## 为什么 Vitalik 说它像预测市场

预测市场的核心逻辑是：

```text
根据某个未来结果，把抵押品分给不同结果 token 的持有人
```

二元预测市场是：

```text
YES + NO = 1
```

事件为真，YES 拿 1，NO 拿 0。

事件为假，YES 拿 0，NO 拿 1。

标量预测市场（Scalar Market）是：

```text
根据一个连续数值，在 long 和 short 之间分配抵押品
```

例如 CPI、气温、价格指数。

Vitalik 的 P/N 也是类似结构：

```text
P + N = 1 ETH
```

到期后根据 `x` 的值分配 ETH。

所以它不是传统意义上的借贷协议，而更像一个标量 payoff 市场。

## 用户如何使用 P

假设ETH当前价格是1500美元，用户希望获得USD稳定敞口；

一个自然策略是持有 `P1000`；

为什么不是 `P1500`？

> 因为 `P1500` 太接近当前价格，如果 ETH 稍微下跌，P 就接近不稳定区域；

`P1000` 更安全，因为当前价格1500距离strike 1000 有较大缓冲；

Vitalik 建议的思路可以简化为：

```text
当前价格 = X
选择 S < X / 2 的 P
选择 1-2 个月后的到期日
如果价格跌到接近 S，比如低于 1.5S，则滚动到更低 S
不要持有到期
```

这个策略的核心是：

> 持有离风险边界足够远的 P，并在接近边界之前主动换仓。

## 再平衡为什么是用户选择

在债务系统中，清算阈值是协议统一规定的；所有用户都暴露在同一类清算逻辑下；

在 P/N 系统中，用户可以选择自己的再平衡策略：

- 保守用户可以频繁滚动；
- 激进用户可以少滚动，接受更多漂移；
- 机构用户可以用自己的 oracle 和风险模型；
- 普通用户可以使用 wrapper 自动化管理；

这带来两个重要好处；

第一，降低 MEV 风险：

如果协议公开规定某个价格必须清算，搜索者可以提前围绕这个阈值进行攻击或套利；

如果用户本地决定何时滚动，外部人不一定知道用户的具体阈值；

第二，减少全局 oracle 依赖：

协议不需要实时使用唯一 canonical oracle 触发全局清算。用户可以用本地 agent、多个数据源或私有策略决定是否提前滚动。

## 再看看N是用来干什么的

`N`的作用可以理解成：它是`P`的另一边，是承接剩余ETH风险和收益的那一方；

`P`主要服务于想要"稳定敞口"的人。比如用户希望自己的资产更像1500美元，而不是直接暴露在ETH波动里；

但系统里只有1ETH，不能凭空创造美元稳定性。所以如果`P`想变得更稳定，必须有人拿走另一部分风险。这个人持有的就是`N`；

举个例子，假设`行权价S = 1500`；

如果到期ETH = 3000美元：

```text
P = 1500 / 3000 = 0.5 ETH
N = 0.5 ETH
```

`N`拿到剩下 0.5 ETH，价值 1500 美元；

如果到期ETH = 2000美元：

```text
P = 1500 / 2000 = 0.75 ETH
N = 0.25 ETH
```

`N`拿到剩下 0.25 ETH，价值 500 美元；

如果到期ETH = 1500美元：

```text
P = 1500 / 1500 = 1 ETH
N = 0 ETH
```

`N`没有剩余的可以拿；

所以 N 的本质是：

> 当 ETH 价格足够高时，`N`拿走`P`不需要的剩余上涨部分；当 ETH 价格下跌接近 strike 时，`N`先承担损失，甚至归零。

因此，`N`通常适合两类人：

- 投机者：看好 ETH 相对稳定或上涨，愿意承担更高风险换取潜在收益；
- 做市商/流动性提供者：为`P`用户提供另一边交易对手，并通过价格、价差或策略收益获得补偿；

也正因为有`N`承接另一边，系统才不需要债务、不需要坏账、不需要清算。`P`不是凭空稳定的，它的稳定性是由`N`承担波动换来的。

## 这个方向的真正难点

Vitalik 的方案不是已经完成的产品。它提出的是一个设计空间，真正难点至少有五个。；

第一，` P `的市场价格不等于简单的到期 payoff，到期前，` P `的价格会包含时间价值、波动率预期和市场流动性因素；

第二，用户需要滚动，如果用户长期持有旧` P `，接近到期或接近 strike 时，稳定性会变差；

第三，滚动有滑点，如果每次滚动成本很高，一年下来可能损失很大；

第四，需要投机者愿意持有` N `，` P `用户想要稳定敞口，另一边必须有人愿意承担相反风险或获得杠杆式 ETH 敞口；

第五，用户体验很复杂，普通用户不想理解 strike、maturity、delta、drift 和 roll。因此需要 wrapper；

## 为什么wrapper可能是创新重点

P/N primitive 本身很优雅，但用户不一定想直接买` P `；

用户真正想要的是：

```text
我存入 ETH 或稳定资产
协议帮我获得相对稳定的 USD 敞口
并且不要突然清算我
```

这就需要上层 wrapper：

- 自动选择合适 strike；
- 自动选择合适 maturity；
- 分散持有多个 P series；
- 定期滚动
- 控制滑点；
- 展示风险指标；

如果底层 P/N 是发动机，wrapper 就是用户能开的车。
